# v4.2 Audit Baseline

Snapshot for the trust-release baseline before any quality work. This note records the main evidence gaps and fragile areas observed in the current codebase.

## Public claims that need evidence

- `README.md:3` and `package.json:4` claim the library is "comprehensive" and covers `430+ functions across 32 categories`, but this repo snapshot does not include a verifiable inventory or count in source. The claim may be true, but it is not evidenced in the audited files.
- `README.md:12-17` claims `full tree-shaking support`, `zero dependencies`, `type-safe`, and an `extensible` plugin system. `package.json:8` and `package.json:140-174` support the export layout and `sideEffects: false`, but there is no explicit tree-shaking proof or bundle-level regression check in the baseline tests.
- `README.md:8` points to a live playground and docs site. That is a public claim outside the repo, so it needs its own external verification if it is part of release trust.
- `README.md:15` describes the library as comprehensive across `32 utility categories`, but the source review shows broad surface area with uneven test depth, especially in timezone and plugin behavior.

## Typed options that are not implemented

- `src/naturalLanguage.ts:12-20` exposes `locale` and `strict` in `NaturalParseOptions`, but neither option is read anywhere in the implementation. The parser currently uses only `referenceDate` and `defaultTime`.
- That means the type surface promises locale-aware and strict parsing behavior that does not exist yet. Callers can pass these options, but they have no effect.

## Fragile architecture areas

- `src/plugins.ts:30-54` depends on a hidden `globalThis.__chainedDateClass` handshake, and `src/chain.ts:496-500` writes that global as side-effect initialization. This creates order dependence between modules and makes the plugin system harder to reason about in tests and in consumer bundlers.
- `src/plugins.ts:105-161` mutates `ChainedDate.prototype` directly and keeps an in-memory registry for rollback. That works for simple cases, but it is global mutable state with collision risk, especially if plugins are loaded/unloaded in different orders.
- `src/timezone.ts:8-20` and `src/timezone.ts:143-187` lean on `Intl.DateTimeFormat` offset parsing and a day-by-day DST scan. Those paths are practical, but they are environment-sensitive and can be brittle around DST boundaries, unusual offsets, and formatter output differences.

## Weakest test surfaces

- `test/plugins.test.ts:18-240` covers the happy path for registration, chaining, and uninstalling, but it does not exercise module-load order, prototype restoration of overwritten methods, or any cross-test contamination scenarios beyond a simple cleanup loop.
- `test/timezone.test.ts:23-155` is broad but shallow. It mainly validates UTC, null-handling, and same-zone behavior. It does not assert a real DST transition date, southern hemisphere behavior, or a non-UTC conversion path with a known expected result.
- The natural-language parser has no dedicated test file in the audited set, so the unused `locale` and `strict` options are not protected by failing tests.

## Baseline expectation

The baseline for this release is expected to stay green on:

- `npm run lint`
- `npm test`
- `npm run build`

This audit note does not change behavior; it only captures the main trust gaps to address next.
